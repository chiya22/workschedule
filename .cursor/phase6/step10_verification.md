# Step 10 — 総合動作確認

Phase 6（認証フロー）全体の総合動作確認を行います。
すべてのフローがエンドツーエンドで動作することを検証してください。

---

## 事前準備

### Supabase の状態をクリーンにする

総合確認の前に、テストデータをクリアしてクリーンな状態から始めます。

```sql
-- すべてのデータを削除（開発環境のみ・本番では絶対に実行しない）
TRUNCATE TABLE notifications, reservations, tables, invitations, staff RESTART IDENTITY CASCADE;
```

Supabase ダッシュボード > Authentication > Users で、テスト用ユーザーも削除してください。

## シナリオ 1: 1人目のオーナー登録フロー

新規にアプリを使い始めるオーナーのフローを確認します。

### 手順

1. ブラウザで `http://localhost:3000` にアクセス
2. `/login` にリダイレクトされる
3. 「こちらから登録」リンクから `/signup` へ遷移
4. メールアドレス・パスワードを入力（招待コードは空のまま）
5. 「アカウントを作成」をクリック
6. 「確認メールを送信しました」と表示される
7. 受信メールの確認リンクをクリック
8. `/onboarding` に遷移
9. 「最初の登録者として、オーナー権限で登録されます」と表示される
10. 名前を入力して「登録を完了する」
11. `/calendar` に遷移してカレンダー画面が表示される
12. ヘッダー右にユーザーメニュー（イニシャル・名前）が表示される

### チェックリスト

- [ ] サインアップ画面で空の招待コードでも送信できる
- [ ] 確認メールが受信できる
- [ ] メールリンクで `/onboarding` に遷移する
- [ ] オンボーディング画面で招待コード欄が表示されない
- [ ] 登録完了で `/calendar` に遷移する
- [ ] Supabase の staff テーブルに `role: 'owner'` のレコードが作成されている

## シナリオ 2: 招待コードによるスタッフ追加フロー

オーナーが新しいスタッフを招待するフローを確認します。

### 手順（オーナー側）

1. ユーザーメニューから「スタッフ管理」をクリック
2. `/settings/staff` に遷移
3. 「スタッフを招待」セクションで権限「スタッフ」を選択
4. 「招待コードを発行」をクリック
5. 表示された招待コード or URL をコピー
6. 「招待中」一覧に追加されていることを確認

### 手順（招待される側 / 別ブラウザ または シークレットウィンドウ）

7. オーナーから受け取った URL でアクセス（`/signup?invite=ABCD2345`）
8. 招待コードが自動入力されている
9. 別のメールアドレスでサインアップ
10. メール確認 → `/onboarding?invite=ABCD2345` に遷移
11. 招待コード欄が表示され、自動入力されている
12. 名前を入力して登録
13. `/calendar` に遷移
14. ユーザーメニューに「スタッフ管理」が表示されない（staff ロールのため）

### 確認（オーナーで戻る）

15. オーナー側で `/settings/staff` をリロード
16. 「登録済みスタッフ」一覧に新しいスタッフが追加されている
17. 「招待中」一覧から該当コードが消えている

### チェックリスト

- [ ] 招待コード発行が成功する
- [ ] コード・URL のコピー機能が動作する
- [ ] 招待コード自動入力が機能する
- [ ] 招待された側がサインアップ → オンボーディング → カレンダーまで遷移できる
- [ ] 一度使った招待コードが accepted ステータスになる
- [ ] スタッフ一覧に新規スタッフが表示される

## シナリオ 3: ログイン・ログアウト

### 手順

1. ログイン状態でユーザーメニューを開く
2. 「ログアウト」をクリック
3. `/login` にリダイレクトされる
4. ブラウザを更新しても自動ログインされない
5. メールアドレス・パスワードを入力してログイン
6. `/calendar` に遷移する

### チェックリスト

- [ ] ログアウトで `/login` にリダイレクトされる
- [ ] ログアウト後、`/calendar` に直接アクセスできない
- [ ] 正しい認証情報でログインできる
- [ ] 間違ったパスワードでエラーメッセージが出る

## シナリオ 4: パスワードリセット

### 手順

1. `/login` 画面で「パスワードを忘れた方」をクリック
2. `/forgot-password` でメールアドレスを入力
3. 「リセットメールを送信」をクリック
4. 「パスワードリセット用のメールを送信しました」と表示
5. 受信メールのリセットリンクをクリック
6. `/reset-password` 画面で新しいパスワードを2回入力
7. 「パスワードを更新」で完了
8. `/calendar` にリダイレクトされる
9. 一度ログアウトして、新しいパスワードでログインできることを確認

### チェックリスト

- [ ] リセット要求のメールが届く
- [ ] リセット画面でパスワード不一致時にエラーが出る
- [ ] パスワード更新後、新しいパスワードでログインできる

## シナリオ 5: アクセス制御の確認

### 未ログイン状態のテスト

ログアウトした状態で以下にアクセス:

- [ ] `/calendar` → `/login?redirect=/calendar` にリダイレクト
- [ ] `/settings/staff` → `/login?redirect=/settings/staff` にリダイレクト
- [ ] `/onboarding` → `/login` にリダイレクト
- [ ] `/login` → アクセス可能
- [ ] `/signup` → アクセス可能
- [ ] `/forgot-password` → アクセス可能

### ログイン済み・staff 未登録のテスト

```sql
-- staff レコードを削除して状態を作る
DELETE FROM staff WHERE user_id = '<YOUR_USER_ID>';
```

ログイン状態で以下にアクセス:

- [ ] `/calendar` → `/onboarding` にリダイレクト
- [ ] `/settings/staff` → `/onboarding` にリダイレクト
- [ ] `/onboarding` → アクセス可能

### スタッフロールでのアクセステスト

- [ ] `/calendar` → アクセス可能

- [ ] `/calendar` → アクセス可能

## シナリオ 6: 期限切れ招待

### 手順

オーナーで招待コードを発行し、SQL で期限を過去に変更:

```sql
UPDATE invitations
SET expires_at = NOW() - INTERVAL '1 day'
WHERE code = '<YOUR_TEST_CODE>'
  AND status = 'pending';
```

新規ユーザーで該当コードを使ってオンボーディング送信:

- [ ] エラーメッセージ「招待コードの有効期限が切れています」が表示される
- [ ] invitations テーブルの status が 'expired' に更新されている

## エッジケース確認

### CSRF / セキュリティ

- [ ] Server Action のフォーム送信が POST メソッドである
- [ ] パスワードがネットワークタブに平文で出ていない（HTTPS時）
- [ ] Service Role Key がクライアントバンドルに含まれていない（DevTools で確認）
- [ ] Cookie に Secure / HttpOnly フラグが立っている（DevTools > Application > Cookies）

### エラーハンドリング

- [ ] ネットワーク切断時にエラーメッセージが表示される（Chrome DevTools の Network タブで Offline にしてテスト）
